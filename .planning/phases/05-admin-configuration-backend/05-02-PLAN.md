**Update Department.java** — add @OneToMany to DepartmentAction join entity so action IDs are accessible via the entity graph. The department_actions table schema (from Flyway V1):
```
department_actions: department_id BIGINT FK→departments.id, action_id BIGINT FK→actions.id, PRIMARY KEY (department_id, action_id)
```
Add to Department.java:
```java
@OneToMany(mappedBy = "departmentId", cascade = CascadeType.ALL, orphanRemoval = true, fetch = FetchType.LAZY)
private List<DepartmentAction> departmentActions = new ArrayList<>();
```
NOTE: DepartmentAction.java already uses composite @IdClass with Long departmentId + Long actionId. The `mappedBy` must reference the field name in DepartmentAction that corresponds to the FK — since DepartmentAction stores departmentId as a plain Long (not a @ManyToOne), use a @OneToMany with a @JoinColumn instead of mappedBy:
```java
@OneToMany(cascade = CascadeType.ALL, orphanRemoval = true, fetch = FetchType.LAZY)
@JoinColumn(name = "department_id")
private List<DepartmentAction> departmentActions = new ArrayList<>();
```
Add getter/setter: `public List<DepartmentAction> getDepartmentActions() {...}` / `setDepartmentActions`.

**DepartmentService.java** (package `com.ureport.admin.service`):

`listDepartments()`:
- departmentRepository.findAllByOrderByNameAsc()
- For each: compute categoryCount via categoryRepository.countByDepartmentId(dept.id) — add this query: `@Query("SELECT COUNT(c) FROM Category c WHERE c.department.id = :id") long countByDepartmentId(@Param("id") Long id);`
- Collect actionIds from dept.getDepartmentActions().stream().map(DepartmentAction::getActionId).collect(toList())
- Return List<DepartmentDetailDto>

`createDepartment(CreateDepartmentRequest req)`:
- Validate name not blank (Jakarta @NotBlank handles this; service can also throw explicitly)
- If defaultPersonId != null → personRepository.findById(defaultPersonId) → throw BusinessException("PERSON_NOT_FOUND","...",404) if absent
- If actionIds not empty → for each id: actionsRepository.findById(id) → throw BusinessException("ACTION_NOT_FOUND","Action not found",404) if absent
- Create Department, set name and defaultPerson
- Save department (get generated ID)
- Reconcile department_actions: for each actionId in req.actionIds, create DepartmentAction(departmentId=dept.id, actionId=actionId) and add to dept.getDepartmentActions() (or save directly via departmentActionRepository)
- departmentRepository.save(department) to cascade
- Return DepartmentDetailDto

`getDepartment(Long id)`:
- departmentRepository.findById(id) → 404 if absent
- Map to DepartmentDetailDto with categoryCount and actionIds

`updateDepartment(Long id, UpdateDepartmentRequest req)`:
- Load department or 404
- Validate/set name, defaultPerson (same checks as create)
- Validate new actionIds all exist
- **Action reconciliation:** Clear existing department_actions via `dept.getDepartmentActions().clear()` (orphanRemoval handles DB delete), then add new DepartmentAction entries for each id in req.actionIds
- departmentRepository.save(department)
- Return DepartmentDetailDto

`deleteDepartment(Long id)`:
- Load department or 404
- If categoryRepository.existsByDepartmentId(id) → throw BusinessException("DEPT_IN_USE","Department has categories and cannot be deleted",HttpStatus.CONFLICT)
- departmentRepository.delete(department) — CascadeType.ALL removes department_actions

`getDepartmentCategories(Long deptId)`:
- departmentRepository.findById(deptId) → 404 if absent
- categoryRepository.findByDepartmentId(deptId) — add to CategoryRepository: `List<Category> findByDepartmentId(Long departmentId);`
- Map to CategoryRef list: `record CategoryRef(Long id, String name, Boolean active) {}`

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→API | Untrusted request body (department name, defaultPersonId, actionIds array) crossing into Spring handler |
| JWT→SecurityContext | JWT cookie decoded into PersonDetails principal for role enforcement |
| API→DB | DepartmentActionRepository delete/save with user-supplied department ID and action IDs |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-07 | Elevation of privilege | `POST /api/departments`, `PUT /api/departments/{id}`, `DELETE /api/departments/{id}` | mitigate | `@PreAuthorize("hasRole('ADMIN')")` on all write methods in DepartmentController; Spring Security method-level enforcement runs before handler body; GET endpoints require JWT but not admin role |
| T-05-08 | Tampering | `PUT /api/departments/{id}` — user-supplied actionIds could reference arbitrary Action records | mitigate | `DepartmentService.updateDepartment` validates every actionId via `actionsRepository.findById(actionId)` before creating DepartmentAction join rows; invalid actionId throws BusinessException ACTION_NOT_FOUND with 404 |
| T-05-09 | Tampering | `DELETE /api/departments/{id}` — cascading delete removes department_actions silently | mitigate | CascadeType.ALL on Department.departmentActions is scoped only to department_actions join table rows, not to categories or tickets; `categoryRepository.existsByDepartmentId` guard prevents deletion of departments with referencing categories and returns 409 DEPT_IN_USE |
| T-05-10 | Information disclosure | `GET /api/departments` and `GET /api/departments/{id}` — department structure is internal configuration | mitigate | JWT required for all `/api/**` routes via SecurityConfig; missing/invalid JWT returns 401 before handler executes |
| T-05-11 | Denial of service | `PUT /api/departments/{id}` — attacker submits actionIds array with hundreds of entries causing bulk DB inserts | accept | Department action lists are small (typically <20 actions); no specific rate limit added in Phase 5 scope. Residual risk owned by infrastructure/ops layer; Spring request size limits (default 2MB) provide a practical upper bound. |
</threat_model>
