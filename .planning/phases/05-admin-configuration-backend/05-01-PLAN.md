**PeopleService.java** (package `com.ureport.admin.service`):

`listPeople(String q, String role, int page, int pageSize)`:
- Call personRepository.searchPeople(q, role, PageRequest.of(page, pageSize))
- Map to PersonPageDto with PersonListItemDto entries
- If q or role are blank/null, pass null to trigger IS NULL branch in JPQL

`createPerson(CreatePersonRequest req, PersonDetails currentUser)`:
- Validate: at least one of firstname, lastname, organization is non-null/non-blank → throw BusinessException("PERSON_NAME_REQUIRED", "...", 400) if all blank
- Username uniqueness: if req.username not null → personRepository.existsByUsername(req.username) → throw BusinessException("USERNAME_CONFLICT", "Username already in use", 409 CONFLICT) if true
- Role guard: if req.role is "admin" or "staff" and currentUser.getRole() != "admin" → throw BusinessException("ROLE_FORBIDDEN", "Only admins may set role to admin or staff", HttpStatus.FORBIDDEN)
- Create Person, set all scalar fields; if departmentId != null load Department or throw BusinessException("DEPT_NOT_FOUND","Department not found",404)
- Reconcile sub-resources: create PeopleEmail/PeoplePhone/PeopleAddress for each item in request lists, link to person via person.getEmails().add(email)
- personRepository.save(person) — cascade saves sub-records
- Return PersonDetailDto mapped from saved person

`getPerson(Long id)`:
- personRepository.findById(id) with explicit @EntityGraph or use Hibernate.initialize() on the collections
- Throw BusinessException("PERSON_NOT_FOUND","Person not found",404) if absent
- Map to PersonDetailDto

`updatePerson(Long id, UpdatePersonRequest req, PersonDetails currentUser)`:
- Load person or 404
- Role guard (same as create)
- Username uniqueness: if changed → existsByUsernameAndIdNot(req.username, id) → 409 if conflict
- Update scalar fields on person
- **Array reconciliation pattern:** For emails: compare req.emails by id. Items with null/absent id → new PeopleEmail added to person.getEmails(). Items with existing id → update the matching entity in person.getEmails(). Items in person.getEmails() whose id does NOT appear in req.emails → remove from collection (orphanRemoval handles delete). Same pattern for phones and addresses.
- personRepository.save(person)
- Return PersonDetailDto

`deletePerson(Long id)`:
- Load person or 404
- Check ticket refs: if ticketRepository.existsByEnteredByIdOrReportedByIdOrAssignedPersonId(id, id, id) → throw BusinessException("PERSON_IN_USE","Person is referenced by tickets and cannot be deleted",HttpStatus.CONFLICT)
- personRepository.delete(person) — CascadeType.ALL removes emails/phones/addresses

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→API | Untrusted request body (person fields, nested email/phone/address arrays) crossing into Spring handler |
| JWT→SecurityContext | JWT cookie from browser decoded into PersonDetails principal |
| API→DB | PersonRepository JPQL queries accepting user-supplied q/role/username parameters |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Elevation of privilege | `POST /api/people` and `PUT /api/people/{id}` — `role` field in request body | mitigate | `PeopleService.createPerson` / `updatePerson` checks `currentUser.getRole() != "admin"` before allowing role=admin or role=staff assignment; throws BusinessException ROLE_FORBIDDEN with HttpStatus.FORBIDDEN; `@PreAuthorize("hasRole('ADMIN')")` on controller methods prevents non-admin access entirely for writes |
| T-05-02 | Tampering | `PersonRepository.searchPeople` — user-controlled `q` parameter in JPQL LIKE clause | mitigate | JPQL named parameters (`@Param("q")`) with `LIKE CONCAT('%',:q,'%')` use parameter binding — Hibernate never performs string interpolation into the query; SQL injection is structurally impossible via JPA param binding |
| T-05-03 | Information disclosure | `GET /api/people` and `GET /api/people/{id}` — returns PII (email addresses, phone numbers, home addresses) | mitigate | JWT required for all `/api/**` routes via SecurityConfig filterChain; missing or invalid JWT returns 401 before handler executes; `passwordHash` field is never included in PersonDetailDto or PersonListItemDto |
| T-05-04 | Elevation of privilege | `DELETE /api/people/{id}` — deleting a person cascades to auth sub-records | mitigate | `@PreAuthorize("hasRole('ADMIN')")` on deletePerson; TicketRepository existence check in `PeopleService.deletePerson` before delete; CascadeType.ALL scope is limited to people_emails/phones/addresses (not ticket records) |
| T-05-05 | Tampering | `PUT /api/people/{id}` — attacker submits array items with fabricated `id` values pointing to other persons' emails/phones | mitigate | `PeopleService.updatePerson` array reconciliation matches items in `person.getEmails()` by id — only IDs already owned by this person's collection are updated; mismatched IDs are treated as new entries (null id), never as cross-person edits |
| T-05-06 | Repudiation | `DELETE /api/people/{id}` — no audit trail | accept | Phase 5 scope is admin configuration APIs; ticket_history covers case-level audit. Person-level audit trail is deferred to Phase 9 admin panels if needed. Residual risk owned by system operator. |
</threat_model>
