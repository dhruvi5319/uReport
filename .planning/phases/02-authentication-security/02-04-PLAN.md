**Create RoleHierarchyConfig at `backend/src/main/java/com/ureport/security/RoleHierarchyConfig.java`.**

From TechArch §5.4: Role hierarchy: ROLE_ADMIN > ROLE_STAFF > ROLE_PUBLIC.

This means an admin-role JWT can access both staff routes and admin routes. A staff-role JWT can access staff routes but NOT admin routes.

```java
package com.ureport.security;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.security.access.hierarchicalroles.RoleHierarchy;
import org.springframework.security.access.hierarchicalroles.RoleHierarchyImpl;
import org.springframework.security.web.access.expression.DefaultWebSecurityExpressionHandler;

@Configuration
public class RoleHierarchyConfig {

    /**
     * Define role hierarchy: ROLE_ADMIN implies ROLE_STAFF implies ROLE_PUBLIC.
     *
     * This means:
     * - ADMIN can access routes restricted to ADMIN, STAFF, or PUBLIC
     * - STAFF can access routes restricted to STAFF or PUBLIC (not ADMIN-only)
     * - PUBLIC can only access routes restricted to PUBLIC (unauthenticated)
     */
    @Bean
    public RoleHierarchy roleHierarchy() {
        return RoleHierarchyImpl.withDefaultRolePrefix()
                .role("ADMIN").implies("STAFF")
                .role("STAFF").implies("PUBLIC")
                .build();
    }

    /**
     * Register the role hierarchy with the web security expression handler
     * so that hasRole() / hasAnyRole() expressions in SecurityConfig and
     * @PreAuthorize annotations respect the hierarchy.
     */
    @Bean
    public DefaultWebSecurityExpressionHandler webSecurityExpressionHandler() {
        DefaultWebSecurityExpressionHandler handler = new DefaultWebSecurityExpressionHandler();
        handler.setRoleHierarchy(roleHierarchy());
        return handler;
    }
}
```

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-19 | Elevation of privilege | Role hierarchy misconfiguration allowing STAFF to access ADMIN routes | mitigate | `AuthorizationIT.deletePerson_staffJwt_returns403()` explicitly verifies that DELETE /api/people/{id} with a staff JWT returns 403. `RoleHierarchyConfig` declares `ADMIN.implies(STAFF)` — NOT the reverse. Test suite prevents regression. |
| T-02-20 | Elevation of privilege | @PreAuthorize annotations on service layer bypassed if not enabled | mitigate | `@EnableMethodSecurity(prePostEnabled = true)` in `SecurityConfig` (02-01) enables `@PreAuthorize`. `AuthorizationIT` loads full `@SpringBootTest` context ensuring the annotation processor is active. `SecurityConfigTest` in 02-01 also validates this. |
| T-02-21 | Tampering | Role claim in JWT modified by attacker to "admin" | mitigate | JWT HS256 signature verified by `JwtService.validateToken()` in `JwtAuthFilter` before role is extracted. Tampered claim invalidates signature → 401 returned. Covered by `SecurityConfigTest.protectedRoute_invalidJwt_returns401()`. |
| T-02-22 | Information disclosure | 403 response leaks existence of admin-only endpoints to staff users | accept | Returning 403 (vs 404) to authenticated staff on admin routes confirms the endpoint exists. Residual risk accepted — endpoint paths are internal API not exposed to public users. Security-through-obscurity is not a substitute for authorization. |
</threat_model>
