**TicketService implementation rules:**

`createTicket(CreateTicketRequest req, PersonDetails currentUser)`:
1. Validate categoryId: load Category; throw 400 CATEGORY_NOT_FOUND or CATEGORY_INACTIVE if missing/inactive
2. Validate location: if req.location is blank AND (req.latitude is null OR req.longitude is null) → throw 400 LOCATION_REQUIRED
3. Build Ticket entity, set status="open", entered_date=NOW(), entered_by_person_id=currentUser.id, last_modified=NOW()
4. Persist ticket via ticketRepository.save(ticket)
5. Create TicketHistory entry: action_id from actionsRepository.findByName("open"), entered_by_person_id=currentUser.id, entered_date=NOW()
6. If req.assignedPersonId != null: update ticket.assigned_person_id, create TicketHistory entry with action_id from actionsRepository.findByName("assignment")
7. Return mapToDto(ticket)

`getTicket(Long id)`:
1. Load ticket with ticketRepository.findById(id) — throw 404 TICKET_NOT_FOUND if absent
2. Load category to get sla_days
3. Compute slaDueDate = ticket.entered_date + sla_days (null if sla_days is 0 or null)
4. Compute isOverdue = (slaDueDate != null) AND Instant.now().isAfter(slaDueDate.toInstant()) AND ticket.status.equals("open")
5. Count mediaCount via mediaRepository.countByTicketId(id)
6. Return mapToDto(ticket) with computed fields

`updateTicket(Long id, UpdateTicketRequest req, PersonDetails currentUser)`:
1. Load ticket; for each non-null field in req, apply to ticket
2. If req.categoryId changed: create "changeCategory" history entry
3. If req.location changed: create "changeLocation" history entry
4. For any other field change: create "update" history entry (only one, even if multiple generic fields changed)
5. Update ticket.last_modified=NOW(), persist, return dto

`closeTicket(Long id, CloseTicketRequest req, PersonDetails currentUser)`:
1. Load ticket; throw 400 TICKET_ALREADY_CLOSED if status=closed
2. Load Substatus by req.substatusId; throw 400 SUBSTATUS_NOT_FOUND if absent
3. If substatus.name = "Duplicate": require req.parentId not null → throw 400 PARENT_REQUIRED
4. Set ticket: status=closed, substatus_id=req.substatusId, closed_date=NOW(), parent_id=req.parentId, last_modified=NOW()
5. Create TicketHistory with action "closed", notes=req.notes
6. If substatus.name = "Duplicate": additionally create TicketHistory with action "duplicate"
7. Persist ticket, return dto

`reopenTicket(Long id, PersonDetails currentUser)`:
1. Load ticket; throw 400 TICKET_NOT_CLOSED if status=open
2. Set: status=open, substatus_id=null, closed_date=null, parent_id=null, last_modified=NOW()
3. Create TicketHistory with action "open"
4. Persist ticket, return dto

`assignTicket(Long id, AssignTicketRequest req, PersonDetails currentUser)`:
1. Load ticket; load Person by req.personId → throw 400 PERSON_NOT_FOUND if absent
2. Set ticket.assigned_person_id=req.personId, last_modified=NOW()
3. Create TicketHistory with action "assignment", notes=req.notes
4. Persist ticket, return dto

**Action lookup:** Inject a private helper `findActionByName(String name)` that calls actionsRepository.findByName(name) and throws an IllegalStateException (500) if the action is missing from seed data — actions table must always have all 10 system actions per Phase 1 seed.

**Repositories needed:** inject TicketRepository, TicketHistoryRepository, MediaRepository, ActionsRepository (new: findByName), CategoryRepository, SubstatusRepository, PersonRepository (existing from Phase 2/3).

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | `TicketController.createTicket` — categoryId, reportedByPersonId, assignedPersonId path | mitigate | Validate all foreign-key IDs server-side via repository lookups in `TicketService.createTicket` before persisting; JPA param-binding (never string interpolation) prevents SQL injection |
| T-04-02 | Elevation of privilege | `POST /api/tickets` and write endpoints | mitigate | `@PreAuthorize("hasAnyRole('STAFF','ADMIN')")` on all write methods in `TicketController`; Spring Security enforces before method body executes |
| T-04-03 | Information disclosure | `GET /api/tickets/{id}` — ticket detail including location/person data | mitigate | JWT required for all `/api/tickets/**` routes per SecurityConfig from Phase 2; missing JWT returns 401 before handler is reached |
| T-04-04 | Tampering | `closeTicket` — parentId can reference any ticket_id | mitigate | Parent ticket existence validated via `ticketRepository.findById(parentId)` in `TicketService.closeTicket` before setting `parent_id`; throws 400 PARENT_NOT_FOUND if absent |
| T-04-05 | Elevation of privilege | `PATCH /api/tickets/{id}` — attacker shifts assignedPersonId to escalate access | mitigate | `@PreAuthorize("hasAnyRole('STAFF','ADMIN')")` restricts assignment patch; person lookup validates target person is an active staff member |
| T-04-06 | Repudiation | Ticket status transitions (close/reopen) — no audit trail | mitigate | Every transition creates an immutable `ticket_history` row via `ticketHistoryRepository.save()` in `TicketService`; rows are insert-only, never updated |
